# ADR（架构决策记录）指南

## 文档目的

记录重要架构与技术决策的背景、选项、结论与后果，便于后续演进与复盘。

## 目标读者

架构师、开发、新成员、审计。

## 单则 ADR 必备结构

1. **标题**  
   简短结论式，如「ADR-001：采用 FastAPI 作为后端框架」。

2. **状态**  
   proposed / accepted / deprecated / superseded（若被后续 ADR 取代，注明 superseded by ADR-xxx）。

3. **背景与问题**  
   要解决的业务或技术问题、约束。

4. **决策**  
   最终选定的方案，一句话概括 + 必要细节。

5. **选项与权衡**  
   考虑过的其他选项及为何不选（可简略）。

6. **后果**  
   正面与负面后果、后续需跟进的行动。

7. **元数据**  
   日期、作者、相关方、相关文档/ADR。

## 组织方式

- 建议每则一文件：`docs/adr/ADR-001-简短标题.md`。
- 维护 `docs/adr/README.md` 索引，按编号与状态列表。

## 不应出现的问题

- 只有结论无背景或选项。
- 与当前架构或 Memory ROOT 中的决策矛盾且未标注 superseded。
- 状态长期为 proposed 且无结论。
