# Claude Code 命令设计（文档治理）

以下命令可在 Claude Code 中配置为 Slash 命令或工作流入口（如 `.claude/commands/` 或 `.cursor/commands/`）。执行前**必须先读取 Memory ROOT**（若存在），再执行对应动作。

---

## /doc-audit

**输入**：可选路径（默认 `docs/` 及仓库根）。  
**动作**：盘点指定范围内文档；对照 document-map 检查类型与存在性；检查元数据、与 Memory ROOT/术语表一致性、交叉引用有效性；输出审计报告（已有、缺失、过时、冲突、建议）。  
**输出**：Markdown 审计报告，含清单与优先级建议。  
**与 Memory ROOT**：以 Memory ROOT 为基准校验项目目标、阶段、技术栈、决策一致性。  
**适用场景**：定期文档健康度检查、发布前文档就绪检查。

---

## /doc-generate

**输入**：用户指定文档类型或名称（如「README」「部署手册」「ADR-002」）；可选：基于的草稿或大纲。  
**动作**：读取 Memory ROOT；若存在已有文档则做增量补全而非覆盖；按 document-map 与对应 *-guide 选择结构与深度；生成或补全内容；写入指定路径；在文末注明相关文档与状态。  
**输出**：生成/更新的文件路径与简短变更说明。  
**与 Memory ROOT**：内容与 Memory ROOT 一致；不足处标记「待补充」。  
**适用场景**：缺口补齐、新项目初始化文档、按模板生成新 ADR/Runbook。

---

## /doc-sync

**输入**：可选文档列表或「全部」；可选：同步维度（术语/交叉引用/版本/状态）。  
**动作**：在选定文档间统一术语（以术语表为准）、修正或补全交叉引用、对齐版本号与状态标记、与 Memory ROOT 对齐表述。  
**输出**：变更文件列表与每份的同步项说明。  
**与 Memory ROOT**：术语与项目状态以 Memory ROOT 为源。  
**适用场景**：术语表或 Memory 更新后批量同步；发布前统一版本与引用。

---

## /doc-update

**输入**：用户指定变更原因（如「架构变更：新增服务 X」「接口 v2 上线」「事故复盘」）及可选受影响文档列表。  
**动作**：根据变更类型确定受影响文档（见 document-map 依赖与同步策略）；读取 Memory ROOT 与现有文档；在已有文档上做增量更新（补章节、改表述、更新引用），不整篇重写除非用户要求；冲突时提示并建议处理。  
**输出**：更新后的文件列表与变更摘要。  
**与 Memory ROOT**：若变更来自 Memory ROOT 更新，以 Memory 为准；否则更新后建议用户同步 Memory（若适用）。  
**适用场景**：需求/架构/接口/发布/事故后的文档跟进。

---

## /doc-gap-analysis

**输入**：可选项目类型（常规/AI 项目）、可选路径。  
**动作**：读取 Memory ROOT；盘点现有文档；按 document-map 与项目类型列出应有文档；标出缺失、仅占位、过旧；给出优先级与依赖顺序建议。  
**输出**：缺口分析报告（表格 + 建议生成顺序）。  
**与 Memory ROOT**：依据 Memory ROOT 判断项目阶段与类型，决定是否包含 AI 专项文档。  
**适用场景**：新接手项目、文档体系规划、迭代前补文档。

---

## /doc-release-check

**输入**：版本号或发布标签（可选）。  
**动作**：检查与发布相关的文档是否就绪：Release Notes/CHANGELOG 是否含该版本、部署手册版本是否更新、API 文档破坏性变更是否说明、文档索引与 README 链接是否有效；可选：跑一遍 quality-checklist 关键项。  
**输出**：通过/不通过项列表与修复建议。  
**与 Memory ROOT**：版本与阶段与 Memory ROOT 一致。  
**适用场景**：发布前文档门禁。

---

## /doc-runbook-create

**输入**：故障现象或告警名称；可选：已有监控/运维文档路径。  
**动作**：基于模板生成单则 Runbook：现象、可能原因、诊断步骤、处理步骤、升级与相关文档；与现有 Runbook 索引衔接。  
**输出**：新建 Runbook 文件路径与索引更新建议。  
**与 Memory ROOT**：服务/组件名与架构、运维文档一致。  
**适用场景**：新告警或新故障类型后的 Runbook 补充。

---

## 命令与 Memory ROOT 的通用约定

- 所有命令执行前：若项目存在 Memory ROOT，先读取并吸收，再执行。
- 生成或更新内容不得与 Memory ROOT 事实性冲突；若用户输入与 Memory 冲突，先提示再按用户选择处理。
- 输出中可建议「是否将本次变更同步到 Memory ROOT」（如架构决策、阶段状态变更）。
