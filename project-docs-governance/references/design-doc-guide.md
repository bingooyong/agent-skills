# 架构与详细设计文档指南

## 1. 系统架构设计文档

**目的**：描述系统整体结构、组件边界、数据流、部署拓扑与关键非功能属性。  
**读者**：架构师、开发、运维、安全。

**必备章节**：
- 概述与设计目标（与项目概述一致）
- 系统边界与上下文图
- 组件/服务列表与职责
- 数据流与关键交互（可配图）
- 部署与运行环境拓扑
- 技术栈与关键依赖（或链接技术选型）
- 非功能考量（可用性、扩展性、安全要点）

**可选**：容量与性能假设、演进与拆分计划。  
**禁忌**：与 Memory ROOT 或技术选型文档冲突；只有图无文字说明；缺少与部署/详细设计的衔接说明。

---

## 2. 详细设计文档

**目的**：支撑开发实现，描述模块、接口、数据结构和关键逻辑。  
**读者**：开发、测试。

**必备章节**：
- 文档范围（模块/子系统的边界）
- 与架构的对应关系（引用架构文档）
- 模块/类/服务划分与职责
- 关键接口（入参、出参、错误；可与 API 文档交叉引用）
- 核心数据流与状态
- 关键算法或业务规则摘要
- 与上下游模块的依赖

**可选**：时序图、状态图、扩展点、配置项说明。  
**禁忌**：大段贴代码替代说明；与 API 文档或数据库设计矛盾；缺少「为何这样设计」的简要理由。

---

## 3. 技术选型说明

**目的**：记录主要技术选型及理由，便于后续演进与审计。  
**读者**：架构、开发、PM。

**必备**：选型项（语言、框架、数据库、中间件等）、选定方案、主要理由、已知权衡与替代方案。版本号建议在配置/环境文档中维护。  
**禁忌**：只列技术名无理由；与架构或 Memory ROOT 中的约束不一致。
