# 错误模式库 Skill

积累项目/团队中反复出现的错误模式及其解决方案，支持按错误特征快速检索，帮助避免重复踩坑。

## 何时使用

- 遇到错误，想查是否有已知模式和解决方案
- 刚解决了一个新错误，想记录下来供后续复用
- 定期审计错误模式库的覆盖率和有效性
- 需要从积累的错误模式生成排查指南

**不适用**：实时错误监控与报警、日志分析系统、生产环境的自动化修复。

## 错误模式分类体系

| 类别 | 说明 | 持久性 | 生命周期 |
|------|------|--------|----------|
| 编译构建错误 | 编译失败、构建配置问题、工具链版本冲突 | 项目级，随工具链稳定后可归档 | 中 |
| 运行时异常 | 空指针、越界、类型错误、未捕获异常 | 跨项目，长期有效 | 长 |
| 性能劣化 | 内存泄漏、慢查询、CPU 飙升、冷启动慢 | 项目级，随优化可降级 | 中 |
| 配置错误 | 环境变量缺失、配置项拼写、多环境不一致 | 项目级，随配置稳定后可归档 | 中 |
| 依赖冲突 | 版本不兼容、传递依赖冲突、幽灵依赖 | 跨项目，随依赖生态变化 | 中 |
| 安全漏洞 | 已知 CVE、注入、权限绕过 | 跨项目，修复后可归档 | 短 |
| 数据不一致 | 数据迁移失败、并发写入、状态不同步 | 项目级，随架构变化 | 中 |
| 集成失败 | 第三方 API 变更、协议不匹配、认证过期 | 项目级，随集成方变化 | 短 |

## 输入

| 输入类型 | 说明 | 必需 |
|----------|------|------|
| 错误现象描述 | 错误信息、stack trace、复现步骤 | 是 |
| 错误上下文 | 技术栈、环境、版本信息 | 是 |
| 操作模式 | 查询 / 记录 / 审计 | 是 |
| 已有错误模式库路径 | 用于匹配和去重 | 否 |

## 输出

- 匹配到的已知模式及其解决方案（查询模式）
- 新记录的错误模式条目（记录模式）
- 模式库审计报告（审计模式）
- 未匹配时的排查建议

## 执行步骤

### 1. 解析错误特征

```
提取错误签名:
  IF 有 error stack trace → 提取顶层错误类型 + 文件路径模式
  IF 有 error code → 提取错误码 + 上下文关键词
  IF 仅有症状描述 → 提取关键名词 + 动词组合

构建搜索关键词:
  错误类型 + 技术栈 + 文件路径模式 + 关键上下文
```

### 2. 匹配已有模式

```
IF 签名完全匹配 → 返回已知解决方案
IF 签名部分匹配 → 提示相似模式，列出差异点
IF 无匹配 → 建议记录为新模式
```

对已有模式库中的每条记录，比较：
- 错误类型和错误码是否一致
- 技术栈是否重叠
- 根因类别是否相同
- 解决方案的适用性

### 3. 记录新模式（记录模式）

用户确认解决后，按以下模板记录：

```markdown
### [错误模式标题]
- **签名**: 错误类型/错误码 + 关键上下文
- **现象**: 可观察的表现
- **根因**: 为什么会发生
- **解决方案**: 具体修复步骤
- **预防措施**: 如何避免再次发生
- **关联模式**: 相关的其他错误模式（如有）
- **出现频率**: 首次 / 偶发 / 频繁
- **记录时间**: YYYY-MM-DD
```

### 4. 审计模式库（审计模式）

对已有模式库进行全面审计：

1. **过期检测** — 模式描述的问题是否已在新版本中修复
2. **重复检测** — 多条记录是否描述同一问题
3. **覆盖检测** — 常见错误类别是否有对应记录
4. **有效性检测** — 解决方案是否仍然适用

输出按优先级排序的治理建议。

### 5. 输出结果

根据操作模式输出对应结果：

**查询模式**：
```markdown
## 错误查询结果

### 匹配模式: [标题]
- 相似度: 高/中
- 解决方案: ...
- 适用条件: ...

### 相似模式（如有差异）:
- 差异点: ...
```

**审计模式**：
```markdown
## 错误模式库审计报告
- 总模式数: N
- 过期: X 条 | 重复: Y 条 | 覆盖缺口: Z 类
- 建议归档: [列表]
- 建议新增: [列表]
```

## 边界与非目标

- **不做**自动修复错误 — 只记录和检索模式，修复由用户执行
- **不做**日志解析和实时监控 — 这是运维工具的职责
- **不做**代码 review — 这是 code-review-checklist 的职责
- **不做**依赖安全分析 — 这是 dependency-analyzer 的职责
- **不做**重构建议 — 这是 refactoring-planner 的职责
- 只管理错误模式知识，不管理错误发生的实时状态

## 验收标准

- 查询模式下每条结果都有相似度评估和适用条件说明
- 记录模式下的新模式条目包含完整的签名、现象、根因、解决方案
- 审计模式输出了过期、重复、覆盖缺口的完整列表
- 没有将一次性、未确认的猜测记录为正式模式
- 匹配逻辑基于签名而非字面文本，能识别语义相似的模式
